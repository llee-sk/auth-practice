## 1. 세션에 저장할 로그인 정보 설계

### 구현할 내용
- [x] 세션에 저장할 로그인 사용자 객체 설계
- [x] 세션 attribute key 상수화

### 확인 내용
- 세션에는 왜 엔티티 대신 별도 DTO를 저장하는가
  > Member 엔티티는 DB 영속성 모델이고, 세션에는 로그인 상태를 식별하는 데 필요한 최소 정보만 있으면 된다.
  > 엔티티를 그대로 세션에 넣으면 비밀번호 같은 불필요하고 민감한 필드까지 함께 보관될 수 있고, JPA 엔티티와 세션 저장 구조가 강하게 엮인다.
  > 그래서 세션 전용 객체인 SessionMember를 만들어 필요한 값만 분리해서 저장한다.
- 세션에 비밀번호 같은 민감 정보를 저장하면 안 되는 이유
  > 로그인 이후 요청을 인증하는 데 비밀번호는 필요하지 않다.
  > 필요 없는 민감 정보를 세션에 저장하면 서버 메모리, 세션 저장소, 로그, 디버깅, 장애 분석 과정에서 노출될 위험만 커진다.
  > 암호화된 비밀번호라도 인증 검증 외에는 사용할 이유가 없으므로 세션에 넣지 않는다.
- 세션 key를 문자열로 흩뿌리지 않고 상수화하는 이유
  > 세션에 값을 저장하고 조회할 때 같은 key를 사용해야 하는데, "LOGIN_MEMBER" 같은 문자열을 여러 곳에 직접 쓰면 오타가 나도 컴파일 단계에서 잡히지 않는다. 
  > 그러면 로그인 정보가 세션에 있는데도 조회하지 못해 미로그인처럼 처리될 수 있다. 
  > 그래서 공통 상수로 분리해서 저장, 조회, 삭제 시 같은 값을 사용하도록 한다.


## 2. 세션 로그인 API 구현
### 구현할 내용
- [x] 기존 로그인 검증 로직 `LoginService.login()`에 로그인 성공 시 세션에 로그인 사용자 정보 저장
- [x] 로그인 성공 후 클라이언트에 `JSESSIONID` 쿠키 생성 확인

### 확인 내용
- 로그인 검증과 세션 저장의 책임 차이
  > 로그인 검증 : 입력받은 이메일/비밀번호가 실제 회원 정보와 일치하는지 확인하는 과정. DB에서 회원을 조회하고 `PasswordEncoder`로 비밀번호를 비교한다.
  >
  > 세션 저장 : 로그인 검증이 성공한 뒤, 이후 요청에서 로그인 상태를 확인할 수 있도록 필요한 사용자 정보만 `SessionMember`로 만들어 `HttpSession`에 저장하는 과정이다.
  >
  > 로그인 검증은 “이 사용자가 맞는가?”를 확인하는 책임이고, 세션 저장은 “이제부터 이 사용자를 로그인 상태로 기억한다”는 책임이다.

- 로그인 성공 시 서버가 `JSESSIONID` 쿠키를 내려주는 흐름
  > 1. 로그인 검증에 성공한 뒤 `session.setAttribute(...)`로 세션에 로그인 사용자 정보를 저장한다.
  >
  > 2. 이때 기존 세션이 없다면 서버의 서블릿 컨테이너가 새 `HttpSession`을 생성하고, 해당 세션을 식별할 id를 만든다.
  >
  > 3. 서버는 응답에 `Set-Cookie: JSESSIONID=...` 헤더를 추가하고, 브라우저는 이 쿠키를 저장한다.
  >
  > 4. 이후 같은 서버로 요청할 때 브라우저는 저장된 `JSESSIONID` 쿠키를 함께 보낸다.

- 세션 쿠키가 이후 요청에서 다시 서버로 전달되는 방식
  > 1. 브라우저는 `JSESSIONID` 쿠키를 저장한 뒤, 같은 도메인과 경로에 해당하는 요청을 보낼 때 HTTP 요청의 `Cookie` 헤더에 자동으로 포함한다.  예: `Cookie: JSESSIONID=...`
  >
  > 2. 서버는 요청의 `JSESSIONID` 값을 이용해 서버 측 세션 저장소에서 기존 세션을 찾는다.
  >
  > 3. 그 세션 안에 `LOGIN_MEMBER` 같은 로그인 사용자 정보가 있으면 로그인된 사용자로 판단하고, 없거나 세션이 만료되었으면 미로그인 상태로 처리한다.
  >
  > 즉, `JSESSIONID`는 “어느 세션인지 찾기 위한 값”이고, 실제 로그인 여부는 그 세션 안에 로그인 사용자 정보가 있는지로 판단한다.

- `HttpSession#setAttribute`, `getAttribute`, `removeAttribute` 사용 방식
  > `setAttribute`는 세션에 데이터를 저장할 때 사용한다. 로그인 성공 시 `LOGIN_MEMBER`라는 key로 `SessionMember`를 저장할 수 있다.
  >
  > `getAttribute`는 세션에 저장된 데이터를 key로 꺼낼 때 사용한다. 현재 로그인 사용자 조회나 인증 확인 시 `LOGIN_MEMBER` 값을 꺼내 로그인 여부를 확인할 수 있다.
  >
  > `removeAttribute`는 세션 전체를 없애는 것이 아니라, 세션 안에 저장된 특정 key의 값만 제거할 때 사용한다.
  >
  > 예를 들어 쇼핑몰 서비스에서 세션 안에 `LOGIN_MEMBER`, `CART_ID`, `RECENT_VIEW_PRODUCT`, `POPUP_CLOSED` 같은 값이 함께 저장되어 있을 수 있다.
  >
  > 이때 `removeAttribute("LOGIN_MEMBER")`를 호출하면 로그인 정보만 삭제되고, 장바구니 id, 최근 본 상품, 팝업 닫기 여부 같은 다른 세션 값은 그대로 유지된다.
  >
  > 즉, "로그인 상태만 풀고 같은 브라우저의 임시 사용 상태는 유지하고 싶을 때" 사용할 수 있다.
  >
  > 반면 `session.invalidate()`는 세션 자체를 무효화하므로 `LOGIN_MEMBER`를 포함해 세션 안의 모든 값이 제거된다.


## 3. 현재 로그인 사용자 조회 API 구현
### 구현할 내용
- [x] 현재 로그인 사용자 조회 API 
  - 로그인되어 있으면 현재 사용자 정보 응답
  - 로그인하지 않았으면 인증 실패 예외 발생
- [x] 미로그인 또는 세션 만료 상황에 사용할 인증되지 않은 사용자용 커스텀 예외 생성

### 확인 내용
- 인증과 인가의 차이
  > 인증(Authentication) : 사용자가 누구인지 확인하는 과정이다.  
  > 예를 들면, 놀이공원 입장을 위해 티켓을 확인하는 과정과 같다.
  >
  > 인가(Authorization) : 인증된 사용자가 무엇을 할 수 있는지 권한을 확인하는 과정이다.  
  > 예를 들면, 티켓 종류에 따라 탑승 가능한 놀이기구가 달라지는 것과 같다.
  >
  > 웹 서비스에서는 로그인 이후 발급받은 세션이나 토큰의 유효성을 검증하고,  
  > 해당 사용자가 접근 가능한 기능과 자원을 결정하는 과정을 의미한다.
  >
  > 또한, 인가는 서비스의 정책과 규칙을 실제 동작으로 적용하는 과정이라고 볼 수 있다.

- 미로그인 상태를 `401 Unauthorized`로 처리하는 이유
  > 미로그인 상태는 인증되지 않은 상태이기 때문에 `401 Unauthorized`로 처리한다.  
  > 즉, 서버가 요청 사용자를 인증할 수 없는 상태를 의미한다.

- 세션 만료와 미로그인 상태를 API 관점에서 어떻게 볼지
  > API 관점에서는 세션 만료와 미로그인 상태 모두 “유효한 인증 정보가 없는 상태”로 볼 수 있다.  
  > 따라서 일반적으로 두 경우 모두 `401 Unauthorized`를 반환한다.
  >
  > - 미로그인 상태 : 처음부터 인증 정보가 없는 상태
  > - 세션 만료 상태 : 기존에는 인증되었지만 현재는 인증 정보가 만료된 상태
  >
  > 즉, 서버 입장에서는 현재 요청 시점에 유효한 인증 정보가 없다는 점에서 동일하게 처리할 수 있다.

- 컨트롤러에서 세션 값을 직접 꺼내는 방식의 장단점
  > 장점
  >
  > 1. `session.getAttribute()`로 바로 값을 사용할 수 있어 구현이 단순하다.
  > 2. 로그인 사용자 정보를 어디서 사용하는지 직관적으로 확인할 수 있어 흐름 이해가 쉽다.
  > 3. 복잡한 인증 구조 없이 간단하게 구현 가능해 소규모 프로젝트에서는 실용적이다.
  >
  > 단점
  >
  > 1. 컨트롤러마다 세션 체크 로직이 중복될 가능성이 있다.
  > 2. 인증 로직과 비즈니스 로직이 섞여 역할 분리가 어려워진다.
  > 3. 인증 방식 변경 시 수정 범위가 커져 유지보수가 어려워질 수 있다.
  >
  > 보완 방법
  >
  > 실무에서는 보통 다음과 같은 방식으로 개선한다.
  >
  > - 인터셉터(Interceptor)
  > - 필터(Filter)
  > - Spring Security
  > - Argument Resolver
  > - `@LoginUser` 같은 커스텀 어노테이션

  
## 4. 로그아웃 API 구현
### 구현할 내용
- [x] 로그아웃 API 구현
- [x] 세션이 없는 상태에서 로그아웃 요청이 들어왔을 때 성공으로 처리

### 확인 내용
- `session.invalidate()`의 의미
  > 현재 `HttpSession`을 무효화한다.
  >
  > 세션이 무효화되면 세션 안에 저장되어 있던 모든 attribute가 제거되고, 기존 `JSESSIONID`로는 더 이상 해당 세션을 사용할 수 없다.

- `removeAttribute`와 `invalidate`의 차이
  > `removeAttribute`는 세션 전체를 없애는 것이 아니라, 세션 안에 저장된 특정 key의 값만 제거할 때 사용한다.
  >
  > 예를 들어 `removeAttribute(SessionConst.LOGIN_MEMBER)`를 호출하면 로그인 사용자 정보만 제거되고, 같은 세션에 저장된 다른 값들은 유지된다.
  >
  > `invalidate`는 세션 자체를 무효화한다. 따라서 `LOGIN_MEMBER`를 포함해 세션 안의 모든 값이 제거된다.

- 로그아웃 API를 멱등하게 만들지 여부
  > 로그아웃 API는 멱등하게 처리한다.
  >
  > 세션이 없거나 이미 만료된 상태에서 로그아웃 요청이 들어와도, 요청 후 로그인 상태가 아니라면 목표를 달성한 것으로 보고 성공 응답을 반환한다.
